# 🚀 自动寻优（Phase A）前后端集成测试反馈与优化建议

**To: 后端研发组**

目前前端已经成功跑通了基于 `CardLayout` 的双轨制视图，并完成了参数热力图、Loss 收敛曲线以及 TopK 数据表的渲染。底层 Grid Search 运转良好，数据流转非常顺畅，辛苦各位！

但在进行全量参数空间（1458 种组合 × 4 次重复 = 5832 次仿真）的压测时，暴露出了几个深水区的逻辑与性能瓶颈。为了保证最终模型的科学性与演示的流畅度，需要后端协助排查并调整以下核心逻辑：

### 🔴 核心问题一：Loss 函数权重倒挂，引发“硬件成本”吞噬“用户体验”

**【现象观察】**
前端渲染出的 TopK 榜单出现了反直觉的排名：

* **Rank 1**: 窗口 13，桌子 23。Loss=0.5303，等待 3.87min，排队峰值 79，座位满载率 77%。
* **Rank 3**: 窗口 13，桌子 45。Loss=0.5729，等待 4.07min，座位满载率降至舒适的 61.6%。

**【底层诊断】**
当前的 `LossEvaluator.java` 权重分配严重失衡。系统为了省下 22 张桌子的“硬件成本分”，宁愿让学生在排队峰值近 80 人、座位满载率极高的极限高压下就餐。

**【Action Item (行动项)】**

1. **引入非线性惩罚机制：** 建议在 `LossEvaluator` 中，当“等待时间”或“排队峰值”突破物理常识阈值（如等待 > 3 分钟，排队 > 50 人）时，采用**指数级惩罚（Exponential Penalty）**，直接让 $Loss$ 爆炸，强制系统淘汰这种“反人类”方案。
2. **开放权重调节：** 请在 `LossConfig.java` 中确保 $\alpha$（成本权重）和 $\beta$（体验权重）是高度解耦的，后续前端会接入滑动条，允许动态传参。

### 🟡 核心问题二：无头模式（Headless）压测耗时过长，存在性能瓶颈

**【现象观察】**
前端在执行全空间搜索时，整体耗时过久。虽然去掉了 UI 渲染，但 5000+ 次完整仿真的 CPU 耗时仍不符合预期。

**【底层诊断】**
怀疑 `SimulationEngine.java` 在无头模式下踩了两个经典的性能大坑：

1. **时间推进机制未切换：** 如果当前仿真依然是基于 `for (int t = 0; t < 86400; t++)` 的固定步长（Tick-based）循环，会有海量的无效空转。
2. **GC (垃圾回收) 击穿：** 5000 次仿真会产生千万级的 `Student` 和 `SimulationEvent` 对象。如果没有做对象复用，会导致严重的 Stop-The-World (STW)。

**【Action Item (行动项)】**

1. **排查引擎步进：** 确认无头模式下是否使用了真正的**事件驱动（Event Priority Queue）**，即直接从当前事件时间戳跳跃到下一个事件时间戳，跳过中间无事件的空白时间。
2. **引入对象池（Object Pool）：** 建议对高频生成的短生命周期对象（如具体的事件类）实现简单的重置复用，大幅降低 GC 压力。

### 🟢 核心问题三：参数三权分立的边界界定

**【现象观察】**
目前系统对哪些参数该写死、哪些该优化的界限不够清晰，容易导致寻优器“作弊”（例如系统偷偷调快了打饭速度来降低排队）。

**【Action Item (行动项)】**
请在 `CanteenConfig` 和 `GridSearchOptimizer` 中严格落实以下参数的权限隔离：

1. **客观环境变量（绝对锁死）：** 总人数、打饭耗时分布、学生步行速度等。优化期间绝对禁止变动。
2. **决策变量（仅开放范围）：** `windowCount` 和 `tableCount`。这是寻优器唯一可以遍历干预的变量。
3. **随机噪声变量（由 Seed 强控）：** 学生到达时间点、是否结伴等。严格通过传入的 `baseRandomSeed` 生成，确保同一组参数的多次 `repeatTimes` 拥有不同的噪声，但每次寻优任务的总噪声具有可复现性。

---

以上三点是目前阻碍我们大屏展示“完美收敛”的最后障碍。确认完毕后请同步，前端这边等待新版接口重测。
