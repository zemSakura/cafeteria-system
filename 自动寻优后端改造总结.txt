食堂仿真系统 - 自动寻优后端改造总结
生成说明：本文件用于记录本次后端自动寻优层改造内容，便于后续展示、答辩和交接。

一、改造目标

本次改造没有推翻原有仿真内核，而是在现有 ArrivalModule 和 SimulationEngine 的基础上新增了一层自动寻优后端能力。

系统现在可以在不启动前端 UI 的情况下，批量尝试不同的窗口数和桌子数配置，对每组参数重复运行多次仿真，并根据等待时间、排队长度、资源利用率、完成率、放弃率和资源成本等指标计算综合 Loss，最终自动推荐一组较优参数。

二、核心设计

新增包：

backend.optimize

主要调用链：

OptimizeMain
  -> GridSearchOptimizer
    -> SimulationAdapter
      -> CanteenConfig
      -> ArrivalModule
      -> SimulationEngine
    -> ResultAverager
    -> LossEvaluator
  -> OptimizationCsvExporter

GridSearchOptimizer 负责遍历参数空间、重复仿真、记录最优结果。
SimulationAdapter 负责把优化参数注入原有仿真系统，并同步运行无头仿真。
LossEvaluator 负责计算综合 Loss。
ResultAverager 负责对同一组参数的多次仿真结果取平均。
OptimizationCsvExporter 负责导出优化结果和回放数据。

   综合 Loss 计算器。

8. ResultAverager.java

13. ReplayRecorder.java

- 原来的 CSV 输出逻辑仍然保留，只是在优化模式下默认关闭。

五、Loss 计算逻辑

Loss 不直接把成本、等待时间和队列长度相加，而是先归一化，再加权求和。

当前主要考虑：

1. 成本
   窗口数和桌子数越多，成本越高。

2. 平均等待时间
   等待时间越短越好。

3. 最大排队人数
   峰值排队越低越好。

4. 放弃率
   放弃就餐人数比例越低越好。

5. 座位利用率
   希望接近目标利用率，太低代表资源浪费，太高代表拥挤。

6. 完成率惩罚
   如果完成率低于最低可接受值，会额外增加惩罚。

六、重复仿真取平均

同一组参数会重复仿真多次。

例如：

窗口数 = 4
桌子数 = 80
repeatTimes = 2

系统会使用不同随机种子运行两次仿真，然后对结果取平均。

不同随机种子会导致学生到达时间、分组、耐心值、窗口选择等随机因素发生变化，因此同一组参数的多次仿真结果可能不同。

取平均的目的：

- 降低随机波动影响。
- 避免某组参数因为一次随机运气好而被误判为最优。
- 避免某组参数因为一次随机特别拥堵而被误判为很差。

十二、答辩描述建议

可以这样描述本次后端架构：

我们没有推翻原有食堂仿真内核，而是在原有 ArrivalModule 和 SimulationEngine 的基础上增加了自动寻优调度层。优化器通过无头模式反复调用原仿真模块，对不同窗口数和桌子数配置进行批量仿真，并收集平均等待时间、最大排队人数、座位利用率、窗口利用率、完成率、放弃率等指标。随后系统通过归一化 Loss 函数综合衡量成本与体验，自动选出最优推荐方案。为了支持后续可视化展示，后端还保留了全过程 allResults 数据，用于绘制 Loss 曲线、当前最优收敛曲线和参数热力图，并在最优参数确定后重新运行一次回放仿真，生成时间序列快照数据。
